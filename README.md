# Network Diagram Editor

ローカル動作のネットワーク構成図エディタです。  
インフラ構成のレビュー資料・教育資料の作成を主な用途として設計しています。

「描けること」より「見て理解しやすいこと」を優先した設計です。  
通信経路・セグメント所属・役割を視覚的に分離することで、第三者が見ても意図が伝わる構成図を手早く作れます。

---

## 動作環境・技術要件

| 項目 | 内容 |
|------|------|
| OS | Windows 10 / 11 |
| Node.js | 18 以上 |
| Rust | 1.70 以上（rustup 経由でインストール推奨） |
| フレームワーク | Tauri v2 + React 18 + TypeScript |
| ビルドツール | Vite 5 |
| 状態管理 | useReducer（React 組み込み） |
| 永続化 | @tauri-apps/plugin-store（ローカルJSON） |
| 外部UIライブラリ | なし（カスタムSVGキャンバス実装） |

---

## セットアップ・起動方法

```bash
# 依存パッケージのインストール
npm install

# 開発モード起動（ホットリロード有効）
npm run tauri dev

# 本番ビルド（インストーラ生成）
npm run tauri build

# Webのみで確認したい場合（Tauri機能は一部非対応）
npm run dev
```

> 初回の cargo ビルドは 5〜15 分程度かかります。  
> データは `%APPDATA%\brainstorming-mvp\` 配下に自動保存されます。

## 主要機能

### ノード (NetworkNode)

| 項目 | 内容 |
|------|------|
| 種別 | server / loadbalancer / database / monitor / user / firewall / router |
| 属性 | 表示名, ホスト名, IPアドレス, 役割, 備考 |
| タグ | セグメントタグ（所属ネットワーク、複数可）/ ユーザータグ（任意） |

**追加方法**
- 左パネルのボタンをクリック → キャンバス中央に追加
- キャンバスをダブルクリック → クリック位置に追加（種別は左パネルで最後に選択したもの）

### 接続線 (NetworkEdge)

| 項目 | 内容 |
|------|------|
| 方向 | 一方向（→）/ 双方向（↔） |
| 通信種別 | 業務通信（青・実線）/ 管理通信（緑・破線）/ 監視通信（紫・点線）/ カスタム |
| プロトコル | SSH / HTTP / HTTPS / PostgreSQL / MySQL / SNMP / ICMP / DNS / NTP / SMTP（複数付与可） |
| ラベル | 自由テキスト |

**作成方法**
1. ツールバーの `Select ON` をクリック
2. 接続元ノードをクリック（番号バッジが表示される）
3. 接続先ノードをクリック
4. `Link →` または `Link ↔` をクリック

### ネットワーク領域 (NetworkArea)

セグメントを背景矩形で表現します。デフォルト名: MGMT / DMZ / APP / DB / MON（カスタム名も追加可能）。

| 操作 | 方法 |
|------|------|
| 追加 | 左パネルのボタンをクリック |
| 移動 | 領域内をドラッグ（中のノードも連動） |
| リサイズ | 右下ハンドルをドラッグ |
| 名前変更 | 領域名をダブルクリック |
| 削除 | 領域名右の `×` ボタン |

### キャンバス操作

| 操作 | 方法 |
|------|------|
| ズーム | マウスホイール |
| パン | 背景をドラッグ、または中ボタンドラッグ |
| グリッドスナップ | ツールバー `Grid ON/OFF` |
| ビューリセット | ツールバー `⌖ Home` |

### ショートカット

| キー | 動作 |
|------|------|
| `Ctrl+Z` | Undo |
| `Ctrl+Y` / `Ctrl+Shift+Z` | Redo |
| ダブルクリック（背景） | ノード追加 |

### 保存・エクスポート

- **自動保存**: 操作のたびにローカルストレージへ自動保存
- **Export JSON**: 図全体をJSONファイルとして書き出し
- **Import JSON**: JSONファイルを読み込み（現在の図は上書き）

```json
{
  "nodes": [
    {
      "id": "...",
      "displayName": "app-01",
      "hostname": "app-01",
      "ipAddress": "10.0.1.11",
      "role": "APサーバ",
      "note": "",
      "nodeType": "server",
      "x": 300, "y": 280, "w": 150, "h": 72,
      "segments": [{ "segmentName": "APP" }],
      "userTags": [],
      "nics": [
        { "id": "...", "name": "eth0", "ipAddress": "10.0.1.11" }
      ],
      "createdAt": 1234567890000
    }
  ],
  "edges": [
    {
      "id": "...",
      "sourceId": "...",
      "targetId": "...",
      "direction": "one-way",
      "protocols": [{ "protocol": "SSH" }],
      "usage": "management",
      "label": "SSH/22",
      "sourceNicId": "...",
      "targetNicId": "..."
    }
  ],
  "areas": [
    {
      "id": "...",
      "name": "APP",
      "color": "#dcfce7",
      "x": 260, "y": 240, "w": 210, "h": 280,
      "nodeIds": ["...", "..."]
    }
  ]
}
```

> `segments` / `protocols` は将来的に構造化属性（NIC名・ポート番号・通信方向等）へ拡張予定のため、現時点でもオブジェクト配列形式で保持しています。  
> `nics` / `sourceNicId` / `targetNicId` は Phase 1 で追加されたフィールドです。現バージョンでは操作UIがなく、Phase 3（React Flow 移行後）で有効化されます。旧データを読み込んだ場合は `nics: []` として扱われます。

## 開発ロードマップ

将来的に React Flow を導入し、以下のフェーズで機能拡張を予定しています。

| フェーズ | ブランチ | 内容 | 状態 |
|---------|---------|------|------|
| Phase 1 | `NDE_phase1_nic_model_addition` | NICデータモデル追加 | **完了** |
| Phase 2 | 未作成 | React Flow 移行（カスタムキャンバス廃止） | 予定 |
| Phase 3 | 未作成 | NIC/ポートUI実装・ポート間エッジ接続 | 予定 |
| Phase 4 | 未作成 | 非矩形エリア・エッジルーティング改善 | 予定 |

**Phase 1 で追加したもの（`NDE_phase1_nic_model_addition`）**

React Flow 移行後に NIC/ポートUIを実装することを見据え、データモデルのみ先行整備しました。画面の見た目に変化はありません。

- `NIC` 型を新設（`id` / `name` / `ipAddress`）
- `NetworkNode` に `nics: NIC[]` フィールドを追加
- `NetworkEdge` に `sourceNicId?` / `targetNicId?` フィールドを追加（将来、エッジを特定ポートに紐づけるために使用）
- Reducer に `addNic` / `updateNic` / `removeNic` アクションを追加（UIは Phase 3 で実装）
- 旧データ互換：保存済みJSONに `nics` がない場合は `[]` で補完

**将来構想**
- ユーザータグを Edge にも付与
- 複数ページ管理
- テンプレート機能（三層Web、LAMP構成 等）
- PNG エクスポート
